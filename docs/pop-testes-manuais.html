01

Objetivo

Padronizar o processo de criação, execução e documentação de testes manuais em todos os projetos de software da ByStartup, garantindo:

  • Rastreabilidade — Cada teste executado tem evidência visual comprovando seu resultado.
  • Consistência — Todos os projetos seguem o mesmo formato e nomenclatura.
  • Qualidade — Bugs são documentados com detalhes suficientes para reprodução e correção.
  • Profissionalismo — O relatório HTML pode ser apresentado ao cliente como prova de qualidade.
02

Escopo de Aplicação

Este POP se aplica a todos os projetos de software da ByStartup, incluindo:

Tipo de Projeto Exemplos Ferramenta de Captura
📱 Aplicativos Mobile React Native, Flutter, nativo (iOS/Android) Screenshot do dispositivo / emulador
🌐 Aplicações Web Next.js, React, Vue, sites estáticos Screenshot do navegador (DevTools)
🔌 APIs / Back-end Node.js, Python, Go Screenshot do Postman / Insomnia / terminal
🖥️ Sistemas Desktop Electron, WPF, etc. Screenshot da janela do app
💡
O relatório é sempre um arquivo .html autocontido — pode ser aberto em qualquer navegador sem necessidade de servidor ou dependências externas.
03

Responsáveis

Papel Responsabilidade
QA / Testador Executa os testes manuais, captura screenshots, preenche o relatório HTML com os resultados e evidências.
Desenvolvedor Mapeia os fluxos do sistema para criação dos casos de teste. Corrige bugs reportados no relatório.
Product Owner / Gerente Revisa e aprova o relatório final. Define os critérios de aceitação para liberação.
Líder Técnico Garante que o POP está sendo seguido e valida a cobertura de testes antes de entregar ao cliente.
04

Pré-requisitos e Ferramentas

🛠️ Ferramentas Obrigatórias

Ferramenta Finalidade Alternativas
Navegador Web Visualizar e validar o relatório HTML Chrome, Firefox, Edge
Editor de Código Editar o template HTML do relatório VS Code (recomendado), Sublime, Notepad++
Captura de Tela Gerar screenshots dos testes Nativo do SO, Lightshot, ShareX
Dispositivo/Emulador Executar o aplicativo em teste Dispositivo físico, Android Studio, Xcode

📋 Pré-condições

  • O projeto deve estar em um ambiente de homologação/staging funcional.
  • Ter acesso ao sistema com credenciais de teste (se necessário login).
  • Conhecer os fluxos principais do sistema (documentados pelo desenvolvedor ou PO).
⚠️
Importante: Nunca execute testes manuais diretamente em ambiente de produção. Use sempre ambientes de homologação ou development.
05

Definições e Glossário

Sigla Significado Descrição
FL-XX Fluxo de Teste Agrupamento lógico de casos de teste por funcionalidade (ex.: FL-01 — Autenticação)
TC-XX Test Case (Caso de Teste) Um cenário específico dentro de um fluxo. Ex.: TC-01 — Login com credenciais válidas
BG-XX Bug Report Registro de um defeito encontrado durante os testes
Aprovado Status ✅ O caso de teste passou — comportamento real = comportamento esperado
Reprovado Status ❌ O caso de teste falhou — encontrado um bug ou desvio do esperado
Bloqueado Status ⚠️ Não foi possível executar o teste por impedimento externo (ambiente fora, dependência)
Pendente Status ⏳ Caso de teste ainda não foi executado

📊 Severidade de Bugs

Severidade Descrição Exemplo
CRÍTICO App crasha, perda de dados, impossível usar funcionalidade core App fecha ao tentar finalizar compra
ALTO Funcionalidade importante com defeito, workaround difícil Popup de erro falso após cadastro concluído com sucesso
MÉDIO Bug visual ou funcionalidade secundária com problema Imagem de produto não carrega em resolução alta
BAIXO Bug cosmético, melhoria de UX, texto com typo Botão desalinhado em telas muito pequenas

⏱️ Prazos, Evidências e Comunicação Obrigatória:
  • Tempo Estimado: Devido à natureza manual, detalhada e minuciosa do processo, a criação, execução e documentação do plano leva de 1 a 3 dias úteis.
  • Evidências Reais: É obrigatório anexar screenshots reais de testes das telas correspondentes aos casos executados. Não utilize mockups ou imagens genéricas.
  • Extensões de Prazo: Esse prazo pode durar mais dias caso sejam encontrados bugs severos (críticos ou altos) que exijam resolução imediata e ciclo de reteste da funcionalidade.
  • Reporte ao Tech Lead: Todo o andamento das execuções, atrasos eventuais causados por bugs graves e bloqueios devem ser reportados diretamente ao Tech Lead da equipe.
Passo 1

🗺️ Mapear os Fluxos do Sistema

Identifique as funcionalidades principais do sistema e agrupe-as em fluxos lógicos. Cada fluxo recebe um código FL-XX.

Exemplo prático:

Código Nome do Fluxo Descrição
FL-01 Autenticação Login, logout, esqueci senha
FL-02 Cadastro Criar conta, validações de campo
FL-03 Busca e Navegação Buscar, filtrar, categorias
FL-04 Compra Carrinho, checkout, pagamento
FL-05 Perfil Editar dados, endereço, senha
💡
Dica: Se o projeto já possui fluxos de teste automatizado (ex.: Maestro, Cypress, Playwright), use-os como base para mapear os fluxos manuais.
Passo 2

📝 Criar Casos de Teste (TCs)

Para cada fluxo, crie casos de teste individuais. Cada TC deve conter:

Campo Descrição Exemplo
Código Identificador único TC-01
Descrição O que está sendo testado Login com credenciais válidas
Pré-condição Estado necessário antes do teste Usuário cadastrado no sistema
Resultado Esperado O que deve acontecer Redireciona para a tela Home
Status Resultado obtido Aprovado ✅ / Reprovado ❌
⚠️
Regra: A numeração dos TCs deve ser sequencial e global em todo o relatório (TC-01, TC-02, ... TC-25), não reinicie a contagem a cada fluxo.
Passo 3

📄 Montar o Relatório HTML

Copie o template base (veja Seção 08) e personalize:

  1. Altere o título e nome do projeto no HTML.
  2. Substitua os fluxos e TCs com os que você mapeou nos passos anteriores.
  3. Configure a variável testData no JavaScript com a quantidade de TCs por fluxo.
  4. Salve o arquivo em docs/relatorio-testes-manuais.html no projeto.
  5. Crie a pasta docs/evidencias/ para armazenar os screenshots.
Passo 4

🧪 Executar os Testes e Capturar Evidências

Para cada TC, siga este processo:

  1. Prepare o cenário — Configure a pré-condição descrita no TC.
  2. Execute a ação — Realize os passos do caso de teste.
  3. Capture a tela — Tire um screenshot do resultado obtido.
  4. Nomeie o arquivo — Use o padrão: tc[número]-[descricao-curta].jpeg
  5. Salve em docs/evidencias/

Padrão de nomenclatura para screenshots:

docs/evidencias/
├── tc01-tela-login.jpeg
├── tc05-minha-conta.jpeg
├── tc07-cadastro-sucesso.jpeg
├── tc11-home-busca.jpeg
├── tc15-detalhes-produto.jpeg
└── tc20-editar-perfil.jpeg
✅
Dica: Capture screenshots em alta resolução e evite cortar informações importantes. Se o teste envolver várias etapas, capture as etapas mais relevantes (início e resultado final).
Passo 5

✅ Registrar Resultados no Relatório

Com os screenshots capturados, atualize o HTML do relatório:

  1. Status do TC: Altere o <span class="status"> para pass, fail, block ou pending.
  2. Evidência visual: Adicione o <img> na galeria do fluxo correspondente.
  3. Observações: Preencha a caixa de observações com detalhes sobre cada TC.
  4. testData: Atualize os contadores no JavaScript (pass, fail, block).

Exemplo — Alterando o status de um TC:

<!-- ANTES (Pendente) -->
<td><span class="status pending dot">Pendente</span></td>

<!-- DEPOIS (Aprovado) -->
<td><span class="status pass dot">Aprovado</span></td>

Exemplo — Adicionando evidência na galeria:

<div class="evidence-item" onclick="openLightbox('evidencias/tc01-tela-login.jpeg')">
  <img src="evidencias/tc01-tela-login.jpeg" alt="TC-01 — Tela de Login"
    onerror="this.style.display='none';" />
  <div class="evidence-caption">TC-01 — Tela de Login ✅</div>
</div>

Exemplo — Atualizando testData:

const testData = {
  auth:     { total: 5, pass: 5, fail: 0, block: 0 },
  signup:   { total: 5, pass: 5, fail: 0, block: 0 },
  search:   { total: 4, pass: 4, fail: 0, block: 0 },
  purchase: { total: 5, pass: 5, fail: 0, block: 0 },
  profile:  { total: 6, pass: 6, fail: 0, block: 0 },
};

│   ├── evidencias/                     ← Pasta com screenshots

│   │   ├── tc05-minha-conta.jpeg
│   │   ├── tc07-cadastro-sucesso.jpeg
│   │   ├── tc11-home-busca.jpeg
│   │   ├── tc15-detalhes-produto.jpeg
│   │   └── ...

💡
Importante: Mantenha os screenshots na pasta evidencias/ com nomes descritivos em kebab-case. O HTML referencia as imagens por caminho relativo (evidencias/nome.jpeg), então ambos devem estar no mesmo diretório pai.
08

Template do Relatório

Para criar um novo relatório, você tem duas opções:

📋 Opção A — Copiar relatório existente

Copie o arquivo relatorio-testes-manuais.html de um projeto concluído e modifique os dados:

  1. Altere o título do projeto na tag <title> e no cabeçalho.
  2. Substitua os fluxos (FL-XX) e casos de teste (TC-XX) com os do seu projeto.
  3. Limpe a galeria de evidências e a pasta evidencias/.
  4. Resete todos os status para pending.
  5. Atualize o testData no JavaScript com os novos totais.

🤖 Opção B — Gerar com Assistente IA (Recomendado)

Utilize o assistente IA (Antigravity / Claude) seguindo este prompt:

Crie um relatório de testes manuais em HTML profissional para o projeto [NOME DO PROJETO].

Os fluxos do projeto são:
- FL-01: [Nome do Fluxo] — [Descrição]
- FL-02: [Nome do Fluxo] — [Descrição]
- ...

Para cada fluxo, crie casos de teste (TC-XX) com:
- Descrição do cenário
- Pré-condição
- Resultado esperado
- Status inicial como "Pendente"

Use o mesmo design visual dark mode do relatório do App Mundo Real.
Inclua: scoreboard com contadores, galeria de evidências, seção de bugs, matriz de cobertura.
O arquivo deve ser salvo em docs/relatorio-testes-manuais.html
✅
A Opção B é recomendada porque o assistente IA já conhece o template e vai gerar um relatório personalizado automaticamente, bastando apenas enviar os screenshots dos testes conforme são executados.

🔧 Campos Customizáveis do Template

Campo Onde editar no HTML Descrição
Nome do Projeto <title> + .topbar-brand span Título exibido na aba e na barra superior
Data de execução .topbar-meta Data em que os testes foram realizados
Fluxos Seções .accordion-item Um bloco por fluxo com tabela de TCs
Casos de Teste Linhas <tr> dentro das tabelas Uma linha por caso de teste
Galeria .evidence-gallery Screenshots por fluxo
Contadores Variável JavaScript testData Totais por fluxo para o scoreboard
09

Registro de Bugs

Quando um TC resulta em Reprovado ❌, registre um bug na seção Log de Bugs do relatório com as seguintes informações:

Campo Descrição Exemplo
ID do Bug Código sequencial BG-01
Título Resumo claro do problema Popup de erro incorreto ao concluir cadastro
Severidade Crítico / Alto / Médio / Baixo ALTO
Fluxo / TC Referência ao caso de teste FL-02 — Cadastro (TC-07)
Passos para reproduzir Passos exatos para reproduzir o bug 1. Abrir cadastro → 2. Preencher → 3. Cadastrar
Resultado Esperado O que deveria acontecer Sucesso sem popup de erro
Resultado Obtido O que realmente aconteceu Popup de erro exibido
Dispositivo / OS Onde o bug foi encontrado Samsung A52 / Android 14

Exemplo de bloco HTML para registro de bug:

<div class="bug-card alto">
  <div class="bug-header">
    <div>
      <span class="bug-id">BG-01</span>
      <div class="bug-title">Popup de erro incorreto ao concluir cadastro</div>
    </div>
    <span class="severity alto">ALTO</span>
  </div>
  <dl class="bug-body">
    <dt>Fluxo</dt>
    <dd>FL-02 — Cadastro de Novo Usuário (TC-07)</dd>
    <dt>Passos para reproduzir</dt>
    <dd>1. Abrir tela de cadastro...</dd>
    <!-- demais campos -->
  </dl>
</div>
🚨
Regra crítica: Todo TC com status Reprovado ❌ deve ter obrigatoriamente um bug registrado na seção de bugs. Não marque um TC como reprovado sem criar o BG correspondente.
10

Critérios de Aceitação

O relatório de testes está aprovado para entrega quando:

Critério Meta Mínima Ideal
Taxa de aprovação geral ≥ 90% 100%
Bugs críticos abertos 0 0
Bugs altos abertos ≤ 1 (com workaround documentado) 0
Cobertura de fluxos 100% dos fluxos core testados 100% de todos os fluxos
Evidências visuais ≥ 1 screenshot por fluxo 1 screenshot por TC
Observações preenchidas Todos os fluxos testados Todos os fluxos
⚠️
Bloqueante: Se a taxa de aprovação for menor que 90% ou houver bugs críticos abertos, o projeto não pode ser entregue ao cliente sem aprovação explícita do PO/Gerente.
11

Checklist Final

Antes de entregar o relatório, verifique cada item abaixo:

📄 Relatório HTML

  • ☐ Título do projeto está correto no cabeçalho e na aba do navegador
  • ☐ Data de execução está atualizada
  • ☐ Todos os fluxos do sistema estão listados
  • ☐ Todos os TCs estão com status preenchido (nenhum "Pendente")
  • ☐ O scoreboard exibe os números corretos (total, aprovados, reprovados)
  • ☐ A barra de progresso reflete a taxa de aprovação real

📸 Evidências

  • ☐ Pasta evidencias/ existe e contém os screenshots
  • ☐ Screenshots estão nomeados de forma descritiva
  • ☐ Cada fluxo tem pelo menos 1 screenshot na galeria
  • ☐ As imagens carregam corretamente ao abrir o HTML no navegador
  • ☐ O lightbox (ampliar imagem ao clicar) funciona

🐛 Bugs

  • ☐ Todo TC "Reprovado" tem um bug correspondente na seção de bugs
  • ☐ Cada bug tem severidade, passos para reproduzir e dispositivo
  • ☐ Bugs corrigidos estão marcados como "Resolvido" com descrição da correção

✅ Validação Final

  • ☐ Relatório abre corretamente no Chrome, Firefox e Edge
  • ☐ Layout está correto em tela cheia e em janela reduzida
  • ☐ Impressão (Ctrl+P) gera um PDF legível
  • ☐ Relatório foi revisado por pelo menos 1 pessoa da equipe
  • ☐ Todos os critérios de aceitação (Seção 10) foram atendidos
