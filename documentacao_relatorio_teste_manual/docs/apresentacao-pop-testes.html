Por que testar manualmente?

🔍
Olho Humano
Detecta problemas visuais, de UX e de fluxo que automação não pega.
📸
Evidências
Screenshots comprovam o funcionamento para o cliente e stakeholders.
🛡️
Confiança
Relatório profissional aumenta a credibilidade da entrega.
🐛
Bugs Reais
Teste no dispositivo real encontra bugs que o emulador não reproduz.
📄

O que é o POP?

Um documento padronizado com 12 seções que guia qualquer membro da equipe a criar relatórios de teste consistentes.

📋 O POP contém:

  • 1️⃣ Objetivo e escopo
  • 2️⃣ Glossário (FL, TC, BG)
  • 3️⃣ Procedimento passo a passo
  • 4️⃣ Template HTML base
  • 5️⃣ Checklist final

🎯 Resultado esperado:

  • ✅ Relatórios padronizados
  • ✅ Evidências visuais
  • ✅ Bugs documentados
  • ✅ Entrega profissional
  • ✅ Rastreabilidade total
🔄

Prazos, Evidências e Comunicação

Diretrizes essenciais de tempo, qualidade e reporte na execução de testes manuais:

📆
Tempo Estimado (1-3 Dias)
O processo de criação, execução e documentação leva de 1 a 3 dias úteis devido à natureza manual minuciosa.
📸
Evidências Reais
É obrigatório anexar screenshots reais das telas testadas. Não utilize imagens genéricas ou mockups temporários.
🐛
Bugs e Extensões
Em caso de bugs severos (críticos/altos) que exijam correção e reteste, o prazo poderá ser estendido.
🗣️
Reporte ao Tech Lead
Qualquer desvio nos prazos, andamento das execuções e impedimentos deve ser reportado diretamente ao Tech Lead.
🛠️

Preparação

Antes de começar, verifique se você tem:

🌐
Navegador Web
Chrome, Firefox ou Edge para visualizar o relatório HTML.
✏️
Editor de Código
VS Code (recomendado), Sublime ou Notepad++ para editar o HTML.
📱
Dispositivo de Teste
Celular físico ou emulador com o app instalado.
📸
Captura de Tela
Ferramenta para screenshots (nativa do SO, Lightshot, ShareX).
⚠️
Nunca execute testes em ambiente de produção. Use sempre homologação/staging.
Passo 1 de 5

🗺️ Mapear Fluxos

Identifique as funcionalidades principais e agrupe em fluxos lógicos (FL-XX):

Código Nome Descrição
FL-01 Autenticação Login, logout, esqueci senha
FL-02 Cadastro Criar conta, validações de campo
FL-03 Busca e Navegação Buscar, filtrar, categorias
FL-04 Compra Carrinho, checkout, pagamento
FL-05 Perfil Editar dados, endereço, senha
Passo 2 de 5

📝 Criar Casos de Teste

Para cada fluxo, defina cenários específicos (TC-XX):

Campo O que preencher Exemplo
Código Sequencial global TC-01
Descrição O que está sendo testado Login com credenciais válidas
Pré-condição Estado antes do teste Usuário cadastrado
Resultado Esperado O que deve acontecer Redireciona para Home
Status Resultado obtido ✅ Aprovado / ❌ Reprovado
⚠️
A numeração é sequencial e global (TC-01, TC-02, ..., TC-25). Não reinicie a contagem em cada fluxo!
Passo 3 de 5

🧪 Executar Testes

Para cada TC, siga este ciclo:

  • 1️⃣ Prepare — Configure a pré-condição descrita no TC
  • 2️⃣ Execute — Realize os passos do caso de teste
  • 3️⃣ Capture — Tire screenshot do resultado
  • 4️⃣ Nomeie — Use o padrão: tc01-descricao.jpeg
  • 5️⃣ Salve — Coloque na pasta docs/evidencias/
docs/evidencias/
├── tc01-tela-login.jpeg
├── tc05-minha-conta.jpeg
├── tc07-cadastro-sucesso.jpeg
├── tc11-home-busca.jpeg
└── tc15-detalhes-produto.jpeg
Passo 4 de 5

✅ Registrar Resultados

Atualize o HTML do relatório com os resultados:

Alterar Status

— Antes (Pendente):
<span class="status pending">
  Pendente
</span>

— Depois (Aprovado):
<span class="status pass">
  Aprovado
</span>

Atualizar Contadores

const testData = {
  auth: {
    total: 5,
    pass: 5,
    fail: 0,
    block: 0
  },
  // ...
};
Passo 5 de 5

🐛 Documentar Bugs

Quando um TC for Reprovado ❌, crie um registro de bug (BG-XX):

Campo Descrição
ID Código sequencial (BG-01, BG-02...)
Título Resumo claro do problema
Severidade CRÍTICO ALTO MÉDIO BAIXO
Passos Passos exatos para reproduzir o bug
Esperado vs Obtido O que deveria acontecer vs. o que aconteceu
Dispositivo Modelo + SO (ex.: Samsung A52 / Android 14)
🚨
Todo TC Reprovado deve ter obrigatoriamente um BG registrado. Nunca reprove sem documentar!

Checklist Final

Antes de entregar o relatório, verifique cada item:

Título do projeto correto no cabeçalho e aba do navegador
Todos os TCs com status preenchido (nenhum "Pendente")
Scoreboard com números corretos
Cada fluxo tem pelo menos 1 screenshot na galeria
Imagens carregam corretamente ao abrir o HTML
Todo TC "Reprovado" tem bug (BG) correspondente
Taxa de aprovação ≥ 90% e zero bugs críticos abertos
Relatório revisado por pelo menos 1 pessoa da equipe
